What is Salt?

Configuration management on the surface. Event-driven everywhere underneath.

Configuration management

Declare your fleet's desired state in YAML. Salt enforces it across thousands of nodes — Linux, Windows, anywhere there's a minion. One source of truth, one apply, every host converges.

saltproject.io ↗

Event-driven by design

Every minion publishes to a real-time event bus. Reactors fire orchestration on those events — no polling, no cron, no five-minute lag. This is what people miss about Salt: it's reactive infrastructure, not just config push.

The default approach

In-place upgrade

  • Database migration runs against your live master
  • Multi-hour change window with the fleet locked out
  • Rollback means restore-from-backup — and pray
  • One failed state and the whole upgrade is in limbo
  • Audit trail is whatever someone screenshotted

The Saltify pattern

Blue/green cutover

  • Stand up a new master alongside the old
  • Copy the certs — minions don't notice the switch
  • Test the new stack under real load before cutover
  • Flip DNS or load balancer when you're happy
  • If something breaks: rebuild, don't restore

Salt 101

How to start, and the commands you'll actually use.

Get started in five steps

01

Install the master

Bootstrap on your control node. The -M flag installs master + minion together so the master manages itself.

# on the master host
curl -fsSL https://bootstrap.saltproject.io | sudo sh -s -- -M
02

Install a minion

Same bootstrap, no flag. Then point it at your master in /etc/salt/minion.

# on each managed host
curl -fsSL https://bootstrap.saltproject.io | sudo sh
echo "master: salt-master.your.domain" | \
  sudo tee /etc/salt/minion.d/master.conf
sudo systemctl restart salt-minion
03

Accept the keys

Minions register with a public key. Inspect, then accept.

# on the master
sudo salt-key -L           # list pending keys
sudo salt-key -A           # accept all pending
04

Wire up git

Treat /srv/salt as a git checkout. A gitpull.service systemd unit lets your CI pipeline trigger a pull over SSH on every promotion.

# pulled by your pipeline on each release
sudo systemctl start gitpull.service
05

Apply your first state

Define a state in /srv/salt/top.sls, then push it to a target.

sudo salt 'web*' state.apply
sudo salt '*'    state.highstate

Essential commands

Connectivity
salt '*' test.ping

Are my minions alive and authenticated?

Apply states
salt '*' state.apply
salt 'db*' state.apply mysql

Run the top file, or apply a specific state to a target.

Inspect facts
salt '*' grains.items
salt '*' pillar.items

Read grains (host facts) and pillar (your data tree).

Run anything
salt 'web*' cmd.run 'systemctl status nginx'

Ad-hoc shell across a fleet. Use sparingly — prefer states.

Local dry run
salt-call --local state.apply test=True

From the minion itself, no master, no commit. Iterate fast.

Watch the bus
salt-run state.event pretty=True

Tail every event on the master. The first thing you open when something's off.

Orchestration
salt-run state.orch orch.deploy_app

Multi-step, multi-host workflows. Blue/green cutovers live here.

Targeting
salt -G 'os:Windows' test.ping
salt -C 'web* and G@env:prod' state.apply

Grain match (-G) and compound (-C) — the targeting you'll use every day.
